# 沦陷度检测指导系统 — AI 工程设计文档

---

## 一、目标定义与验收指标

### 1.1 产品目标

将三套亲密关系理论（猜疑链动力学 / τ自指循环周期 / 关系力学全景）整合为一个**统一的、渐进式的诊断与指导系统**。

**一句话定义**: 亲密关系的实时诊断仪 + 行动导航仪。

### 1.2 核心用户流

```
描述关系 → 系统部署探测任务 → 用户执行并汇报 → 系统诊断 + 指导
     ↑                                                    │
     └──────────── 多轮迭代，画像越来越精确 ──────────────┘
```

### 1.3 验收指标

| 指标 | 定义 | 通过阈值 |
|------|------|---------|
| 理论覆盖度 | 三套理论的核心变量在系统中均可被探测和展示 | 100% |
| 流程完整性 | input→mission→report→result→dashboard 全链路可跑通 | 全部通过 |
| 模式切换 | 3种诊断深度可自由切换，共享关系背景和历史数据 | 无数据丢失 |
| AI输出解析率 | JSON解析成功率（含重试） | ≥ 95% |
| 首次可用时间 | 从打开到完成第一轮诊断 | < 3分钟 |
| 移动端适配 | 460px宽度下所有元素正常显示和交互 | 无溢出/截断 |

## 二、方案选型（决策树走查）

### 2.1 是否需要 AI？

```
任务: 根据用户描述的关系现状，生成个性化探测任务和话术，
      根据对方反应做出精准诊断

→ 需要"理解"自然语言 ✓
→ 需要"生成"个性化内容 ✓
→ 准确率可容忍误差（不是医疗/法律场景）✓
→ 结论: 纯 AI 方案可行
```

### 2.2 API vs 自托管

```
→ 个人项目，数据隐感度低 ✓
→ 日请求量 < 100 ✓
→ 需要顶级语言理解和生成能力 ✓
→ 结论: API 方案（Claude Sonnet）
```

### 2.3 RAG vs 微调 vs 提示工程

```
→ 理论框架是固定的知识（不会频繁更新）
→ 核心挑战是让模型"按框架思考"，不是查找信息
→ 理论文档总量 < 5000 字，可完整放入 system prompt
→ 结论: 纯提示工程，将理论框架编码进 System Prompt
```

### 2.4 单 Prompt vs 多 Prompt

**决策点**: 3种模式是否共用一个 System Prompt？

| 方案 | 优点 | 缺点 |
|------|------|------|
| 单一大 Prompt | 一致性强，维护简单 | Token 消耗高，可能注意力稀释 |
| 3个独立 Prompt | 每个更精准，token 更省 | 维护3份，理论一致性风险 |
| 1个基础 + 模式修饰符 | 平衡一致性和精准度 | 实现稍复杂 |

**决策: 方案三 — 1个基础 Prompt + 模式修饰符**

理由:
- 基础 Prompt 包含统一理论框架（~1200 tokens）
- 模式修饰符仅追加该模式的输出格式要求（~300 tokens）
- 总消耗 ~1500 tokens，可接受
- 理论一致性有保障

### 2.5 模型选择

| 模型 | 价格(input/output) | 质量 | 延迟 | 建议 |
|------|-------------------|------|------|------|
| Claude Sonnet 4 | $3/$15 per 1M | 高 | 中 | **默认选择** |
| Claude Haiku 4.5 | $0.8/$4 per 1M | 中高 | 快 | 快速模式备选 |
| GPT-4o-mini | $0.15/$0.6 per 1M | 中 | 快 | 成本敏感备选 |

**决策: Claude Sonnet 4 为默认**，理由：
- 需要深度理解复杂理论框架并准确应用
- 需要生成高质量中文话术
- 个人项目，成本不是首要考量

## 四、Prompt 工程设计

### 4.1 基础 System Prompt（所有模式共享）

```
你是亲密关系诊断智能体，整合三套理论框架：

【框架一·猜疑链动力学】
爱情势能 E = S(猜疑张力) × I(不可逆浓度) × Δ(变化率)。
S = 1-M/T，健康区间0.3~0.7。I = Σ(w_k·c_k)/N，健康区间0.1~0.3。
Δ = R/P（重建/打补丁比），Δ>1为健康。三变量任一归零E归零。
I_target = 0.2×(1+(0.5-S))，S和I形成跷跷板。

【框架二·τ自指循环周期】
τ(x,D) = x从被提出到被反弹的时间。权力 P(A,B) = τ(B→A)/τ(A→B)。
关系动力学：dF_R/dt = λ(D_want·D_feedback - F_R) - γ·Δτ²。
阶段特征：初期τ双短对称→稳定期τ延长对称→危机期τ不对称。
修复=同步τ节奏，非增加信息量。意向梯度I_n每增一级复杂度指数增长。

【框架三·关系力学全景】
四维关系空间ℛ: Q_Ⅰ身体场 Q_Ⅱ制度场 Q_Ⅲ直觉场 Q_Ⅳ价值场。
自指守恒律：Σ_stage + Σ_debt = Σ_total，不做阶段消化则债务积累至崩溃。
θ_OC = F_创造/F_秩序，需动态平衡，不可同源控制。
纵向因果C_V：高层级的什么结构蕴含了低层级现象的必然出现。

【统一原则】
- 所有诊断归结为纵向因果，不停在横向表象
- 话术必须可直接复制发送
- 风格：军师式，犀利精准，不模糊安慰
- 每个探测任务精确针对一个变量
```

### 4.2 模式修饰符

**快速模式 (E = S×I×Δ)**:
```
当前为快速模式。聚焦 S(猜疑张力)、I(不可逆浓度)、Δ(变化感知)三维度。
输出侧重：直觉判断、沦陷等级、信号强度。
```

**标准模式 (τ测度)**:
```
当前为标准模式。聚焦 τ动态：τ(她→你)、τ(你→她)、D_want、D_feedback、I_n。
输出侧重：τ不对称诊断、权力比、同步建议。
```

**深度模式 (关系力全景)**:
```
当前为深度模式。六维诊断 + 四象限 + 纵向因果。
输出侧重：四象限分布、因果分层(C_H→C_V)、自指债务、秩序/创造平衡。
```

### 4.3 任务生成 Prompt 模板

```
关系背景：{context}
{历史数据（如有）}
{当前模式修饰符}

根据当前数据缺口，选择最需要探测的变量，部署任务。

严格JSON返回：{
  "probe": "变量代码",
  "probe_name": "变量中文名",
  "intent": "战术意图12字内",
  "lines": [{"text": "可直接发送的话术", "when": "发送时机"}],
  "action": "具体行动指令",
  "watch": "精确观察指标",
  "note": "注意事项"
}
```

### 4.4 评估 Prompt 模板（按模式分）

每种模式的评估输出 JSON 不同，但共享部分字段：

**共享字段**:

### 4.5 Prompt 工程风险与对策

| 风险 | 概率 | 对策 |
|------|------|------|
| JSON 输出格式错误 | 中 | 1.明确"严格JSON" 2.正则清洗 3.二次重试 |
| 话术质量不稳定 | 低 | Prompt 中强调"可直接复制发送" |
| 理论应用偏离 | 低 | 基础Prompt中已编码核心公式和健康区间 |
| 评分过于集中(50-60) | 中 | Prompt 中要求"基于具体行为证据打分，不要凑整数" |
| 探测变量重复 | 中 | 传入历史数据，提示"根据数据缺口选择" |

## 五、数据模型与状态流

### 5.1 核心状态

```javascript
// 关系档案（跨模式共享）
profile = {
  context: string,        // 关系背景描述
  createdAt: timestamp,   // 创建时间
}

// 当前会话状态
session = {
  mode: "quick" | "standard" | "deep",  // 当前模式
  step: "input" | "mode" | "mission" | "report" | "result" | "dashboard" | "history",
  mission: MissionObject | null,
  feedback: string,
  result: ResultObject | null,
  loading: boolean,
  error: string,
}

// 累积数据（跨轮次、跨模式）
cumulative = {
  rounds: [{
    mode: string,
    probe: string,
    task: string,
    feedback: string,
    result: ResultObject,
    timestamp: number,
  }],

  // 聚合指标（每轮更新后重新计算）
  metrics: {
    F_R: number,          // 最新关系力
    phase: string,        // 最新阶段
    // 快速模式指标
    S: number, I: number, D: number,
    // 标准模式指标
    tau_her: number, tau_you: number, delta_tau: number,
    D_want: number, D_feedback: number, I_n: number,
    // 深度模式指标
    Q1: number, Q2: number, Q3: number, Q4: number,
    sigma_debt: number, theta_OC: number, P_ratio: string,
    C_V: string,
  }
}
```

### 5.2 状态流转

```
      ┌───── restart ──────────────────────────────────┐
      │                                                │
      ▼                                                │
   [input] ──→ [mode] ──→ [mission] ──→ [report] ──→ [result]
                  ▲              │           │            │
                  │              └── back ───┘            │
                  │                                      │
                  └──── next round (auto-select mode) ───┘
                                     │
                                     └──→ [dashboard] (any time)
                                     └──→ [history] (any time)
```

### 5.3 指标聚合策略

每轮结束后，`cumulative.metrics` 更新规则：
- **直接覆盖**: F_R, phase, C_V（取最新值）
- **加权平均**: S, I, D, tau_*, D_want 等数值指标（最近3轮加权 0.5/0.3/0.2）
- **取最新非空**: Q1-Q4, sigma_debt, theta_OC（仅深度模式产出）
- **趋势计算**: 从 rounds 数组计算 F_R 趋势线

1. **渐进揭示**: 默认显示核心指标，详细数据折叠或下钻

## 七、成本估算与 Trade-off

### 7.1 Token 消耗估算

| 环节 | Input tokens | Output tokens | 说明 |
|------|-------------|---------------|------|
| 任务生成 | ~1800 | ~300 | system(1200) + user(600) |
| 结果评估 | ~2200 | ~500 | system(1200) + user(1000) |
| **单轮总计** | **~4000** | **~800** | |

### 7.2 成本估算 (Claude Sonnet 4)

| 使用频率 | 月轮次 | 月费用 | 备注 |
|---------|-------|--------|------|
| 轻度 (2轮/天) | 60 | ~$0.36 | 探索阶段 |
| 中度 (5轮/天) | 150 | ~$0.90 | 日常使用 |
| 重度 (10轮/天) | 300 | ~$1.80 | 密集期 |

**结论**: 成本极低，无需优化。

### 7.3 延迟预估

| 操作 | 预估延迟 | 优化空间 |
|------|---------|---------|
| 任务生成 | 2-4s | 可用 streaming |
| 结果评估 | 3-6s | 可用 streaming |

### 7.4 关键 Trade-off

| 决策点 | 选择A | 选择B | 决策 |
|--------|------|------|------|
| Prompt 大小 vs 理论完整性 | 精简Prompt省token | 完整理论保质量 | **B**: 质量优先，成本可忽略 |
| 单文件 vs 模块化 | 单JSX文件易部署 | 多文件易维护 | **A**: 个人项目，与现有模式一致 |
| 实时评估 vs 批量评估 | 每轮实时AI评估 | 攒几轮批量评估 | **A**: 即时反馈更重要 |
| 数据持久化 | 仅会话内(state) | localStorage | **两阶段**: MVP用state，后续加localStorage |
